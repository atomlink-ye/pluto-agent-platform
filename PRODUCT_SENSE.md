# PRODUCT_SENSE.md

## Purpose

This document defines product judgment and prioritization rules.

## Primary product bet

The product is valuable if it turns agent activity into an operator-visible, governed, repeatable execution system.

## Scope judgment rules

### Prefer the minimum stable core

When choosing between a smaller coherent capability and a larger diffuse one, prefer the smaller coherent capability.

### Prefer governed execution over feature breadth

Approvals, artifacts, visible state, and replay matter more than wide surface-area ambition.

### Prefer clarity over generality

If a feature broadens the platform but weakens the clarity of playbook, harness, or run semantics, it should wait.

### Prefer operator value over novelty

The system should help someone understand and steer a real run. Fancy orchestration that is hard to observe is a bad trade.

## V1 bias

V1 should favor:

- one strong run lifecycle
- one clear operator experience
- one clear approval path
- one clear artifact path
- one clear evaluation discipline

V1 should defer:

- broad enterprise administration
- heavy multi-surface parity
- over-generalized orchestration modes
- advanced analytics before basic observability works
